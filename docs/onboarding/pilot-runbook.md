# Pilot Runbook

## Purpose

This runbook describes how to run a controlled first adoption of the public DevSecOps Governance Framework in an application repository.

The pilot should prove the integration path, evidence flow, and decision process before any governance check becomes blocking.

## Pilot Outcome

At the end of the pilot, the application team should have:

| Outcome | Description |
| --- | --- |
| Working pipeline integration | The application repository calls the public reusable governance workflows. |
| Evidence artifacts | The pipeline uploads application, security, governance, and optional architecture evidence. |
| Reviewed findings | Findings are understood, triaged, and assigned. |
| Adoption decision | A decision record states whether to stay in `report-only`, start a controlled blocking pilot, or stop. |
| Follow-up owner | A named owner is responsible for evidence quality and recurring maintenance. |

## Roles

| Role | Responsibility |
| --- | --- |
| Application owner | Owns the application repository and pilot decision. |
| Pipeline owner | Adds workflows, evidence generation, and artifact upload. |
| Security reviewer | Reviews SBOM, vulnerability, static-analysis, and waiver evidence. |
| Architecture reviewer | Reviews architecture evidence when architecture governance is in scope. |
| Governance reviewer | Checks whether the result is ready for `report-only` continuation or controlled blocking. |

## Prerequisites

- The application repository can run GitHub Actions.
- The application repository has a default branch and pull request workflow.
- The team can produce or approximate first-run evidence.
- Branch protection expectations are understood before blocking mode is considered.
- The team agrees that placeholders are allowed only for first wiring tests.

## Step 1: Select Pilot Scope

Choose a repository that is representative but not release-critical for the first run.

Record:

- repository name,
- application owner,
- pilot start date,
- expected pilot duration,
- whether architecture governance is in scope,
- expected decision date.

## Step 2: Copy The Adoption Package

Copy the DevSecOps workflow:

```text
adoption-package/workflows/devsecops-baseline.yml
```

to:

```text
.github/workflows/devsecops-baseline.yml
```

When architecture governance is in scope, also copy:

```text
adoption-package/workflows/architecture-governance.yml
```

to:

```text
.github/workflows/architecture-governance.yml
```

Use the first adoption checklist:

```text
adoption-package/checklists/first-adoption-checklist.md
```

## Step 3: Keep First Runs Report-Only

The first pilot runs should not block pull requests or releases.

Expected first mode:

```text
report-only
```

Blocking mode may be considered only after:

- evidence paths are stable,
- findings are understood,
- branch protection is configured,
- waiver handling is agreed,
- owners accept the maintenance effort.

## Step 4: Replace Placeholder Evidence

Replace placeholder files with application-specific evidence where possible:

| Evidence | Pilot expectation |
| --- | --- |
| Application artifact | Real build artifact, package metadata, or source package. |
| SBOM | Generated by an agreed SBOM tool. |
| Vulnerability scan | Generated by an agreed scanner. |
| Static analysis | Linked or summarized when available. |
| Governance run input | Generated from real repository and pipeline context where possible. |
| Architecture evidence | Owned and reviewed when architecture governance is in scope. |

It is acceptable to start with placeholders only to prove wiring. Placeholders must be listed as limitations in the decision record.

## Step 5: Run The Pilot

Run the workflow through:

1. manual `workflow_dispatch`,
2. pull request,
3. protected branch or integration branch push when appropriate.

Collect:

- workflow URLs,
- produced artifacts,
- job summaries,
- failed or warning controls,
- missing evidence,
- required repository settings.

## Step 6: Review Results

Review findings in three groups:

| Group | Meaning |
| --- | --- |
| Wiring issue | Workflow, path, permission, artifact, or reusable workflow reference is wrong. |
| Evidence issue | Evidence is missing, placeholder-only, stale, or not generated by the expected tool. |
| Governance issue | The result indicates a real policy, control, security, architecture, or process gap. |

Do not treat a green `report-only` run as production approval. It means the integration is technically working.

## Step 7: Decide Next Mode

Use the adoption decision record template:

```text
adoption-package/templates/adoption-decision-record.md
```

Recommended decision states:

| State | When to use |
| --- | --- |
| Continue report-only | Wiring works, but evidence or findings are not mature enough for blocking. |
| Controlled blocking pilot | Evidence is stable and selected checks can become required in a limited scope. |
| Not ready | Blocking issues prevent meaningful adoption until resolved. |

## Step 8: Follow Up

Create follow-up actions for:

- missing or weak evidence,
- repository protection gaps,
- scanner configuration,
- waiver process,
- architecture evidence ownership,
- documentation updates,
- baseline or template improvements.

## Minimum Acceptance Criteria

A pilot is successful when:

- the public baseline workflows execute from the application repository,
- required artifacts are uploaded,
- findings can be reviewed by the owning team,
- limitations are visible,
- the adoption decision is documented,
- next steps have owners.

## Stop Criteria

Pause the pilot when:

- the application repository cannot call the public reusable workflows,
- evidence cannot be produced or uploaded,
- required repository permissions are not available,
- findings cannot be reviewed by accountable owners,
- legal, licensing, or security review raises unresolved concerns.
